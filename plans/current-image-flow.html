A Simple Pattern

Config: include_cwd = true, no keyframe system. Uses offload.toml directly.

Example: sculptor unit tests, Offload self-tests (offload-cargo-modal.toml)

cached Runs once, reused across runs
every run Rebuilt on every offload run
parallel Concurrent with other operations

Image Layer Stack

Final Image (not cached)
base + source overlay + sandbox_init_cmd = sandbox-ready image
sandbox_init_cmd layer
e.g. uv sync --all-packages -- baked in at build time
Source overlay (every run)
add_local_dir(".", "/app") -- entire working directory
Base Dockerfile image (cached in .offload-image-cache)
System deps, toolchain, language runtimes

Flow Steps

1
▶ offload run once
Rust CLI loads config, dispatches to Modal provider
  • Loads offload.toml (or -c <path>)
  • Reads [provider] type = "modal" with dockerfile = ".devcontainer/Dockerfile" and include_cwd = true
  • Dispatches to ModalProvider::from_config()
↓
2
▶ ModalProvider::from_config() once
Rust builds the prepare command string for Python
Constructs the shell command:
uv run @modal_sandbox.py prepare .devcontainer/Dockerfile --include-cwd --cached
  • Appends --sandbox-init-cmd=<quoted cmd> if offload.sandbox_init_cmd is set
  • Appends any --copy-dir=local:remote pairs from config or CLI
  • The @modal_sandbox.py reference is resolved to the bundled script cache
↓
3
▶ Python prepare command every run
Builds the full image: base + source + init. This is the expensive step.

Base image (cached):

  • Reads .offload-image-cache for cached base image ID
  • Cache hit: modal.Image.from_id(cached_id) -- skips Dockerfile rebuild
  • Cache miss: modal.Image.from_dockerfile(path, context_dir=".") then .build(app), creates temp sandbox to materialize, writes ID to .offload-image-cache

Source overlay (NOT cached -- runs every time):

  • add_local_dir(".", "/app") -- uploads entire working directory
  • Respects .dockerignore patterns for filtering
  • This is where current source code enters the image

Extra directories:

  • Any --copy-dir directories added via add_local_dir(local, remote)

Init command:

  • sandbox_init_cmd runs via run_commands() (e.g. uv sync --all-packages)
  • This is baked into the image layer, NOT run at sandbox creation time

Expired image handling:

  • If cached base image was GC'd by Modal: catches exception, clears .offload-image-cache, rebuilds from scratch

Output: prints final image_id to stdout

↓
4
▶ Rust captures image_id once
Stores in ModalProvider.image_id for sandbox creation
  • Reads last line of stdout from the Python process
  • Stores as ModalProvider.image_id
  • All subsequent sandbox creation uses this single image ID
↓
5
▶ create_sandbox() called N times parallel
Each sandbox is an isolated instance of the image
uv run @modal_sandbox.py create --cpu 0.125 <image_id>
  • Returns a sandbox_id for each sandbox created
  • Sandboxes are pooled up to max_parallel
  • Each sandbox gets an identical copy of the final image
↓
6
▶ Tests execute on sandboxes parallel
Exec commands sent to sandboxes, JUnit XML collected
  • Batches of tests distributed via LPT scheduling (or round-robin)
  • Each batch runs in its own sandbox via exec_command
  • JUnit XML results downloaded from each sandbox
  • Results aggregated, retries dispatched for failures
Key Observation

Source code is re-uploaded on every single run via add_local_dir(".", "/app"). The base Dockerfile image is cached, but the final image (with source + sandbox_init_cmd) is rebuilt every run. This is the primary bottleneck that the checkpoint-images feature aims to address.

B Keyframe Pattern

Config: include_cwd NOT set. Uses .offload-base-commit + justfile orchestration.

Example: mng tests, sculptor integration tests

cached Baked into Dockerfile layer
every run Rebuilt on every run
manual Requires human intervention to update

Image Layer Stack

Final Image (rebuilt each run)
Dockerfile layers + patch applied + deps re-synced
sandbox_init_cmd (every run)
git apply /offload-upload/patch --allow-empty && uv sync --all-packages
Patch layer (every run, small)
--copy-dir="/tmp/<uuid>:/offload-upload" -- just the git diff
Source at pinned SHA (in Dockerfile, cached)
COPY . /code/ + RUN tar -xzf current.tar.gz + RUN uv sync
Base Dockerfile (cached)
System deps, toolchain, uv, claude code

Flow Steps

Click any step to expand details.

1
▶ justfile target test-offload every run
Orchestrates tarball creation, patch generation, and offload invocation

1a. Read pinned SHA

  • Reads .offload-base-commit file (contains a git SHA like 73ce8f3cc81924aba41ecf000524975b18979f6a)
  • This file is checked into the repo and manually updated by a developer

1b. Create tarball manual sha

  • Runs make_tar_of_repo.sh <sha> <tmpdir>
  • Clones current repo to temp dir
  • Checks out the pinned SHA
  • Creates current.tar.gz tarball of repo at that commit
  • Copies tarball to CWD (where Dockerfile can see it)

1c. Generate patch every run

  • Runs generate_patch_for_offload.sh <sha>
  • Executes git diff <sha> (against working tree -- uncommitted changes included)
  • Outputs patch to /tmp/<uuid>/patch

1d. Invoke Offload

offload -c offload-modal.toml run --copy-dir="/tmp/<uuid>:/offload-upload"
↓
2
▶ Dockerfile builds base image cached
Dockerfile contains pinned-SHA source code and all dependencies

The Dockerfile (e.g. libs/mng/imbue/mng/resources/Dockerfile) does heavy lifting:

  • Installs system deps (git, curl, build tools, etc.)
  • Installs uv, claude code
  • COPY . /code/ -- copies the tarball and Dockerfile context into image
  • RUN tar -xzf /code/current.tar.gz -C /code/mng/ -- extracts pinned-SHA repo
  • RUN uv sync --all-packages -- installs Python deps (BAKED INTO IMAGE)

Source code at the pinned SHA + all deps are in this Docker layer. This is the "keyframe" -- a known-good state.

↓
3
▶ modal_sandbox.py prepare every run
Same as Pattern A but WITHOUT add_local_dir. Source is already in Dockerfile.
  • include_cwd is false, so NO add_local_dir(".", "/app")
  • Source code is already in the Dockerfile layer from step 2
  • The patch is passed via --copy-dir="/tmp/<uuid>:/offload-upload"
  • sandbox_init_cmd applies the patch:
git apply /offload-upload/patch --allow-empty && uv sync --all-packages
  • The uv sync re-runs to pick up any dependency changes in the patch
  • Only the git diff patch is transferred, not the whole repo
↓
4
▶ Sandbox creation and test execution parallel
Same as Pattern A from here on
  • Rust captures image_id from prepare stdout
  • Creates N sandboxes from the image
  • Distributes test batches, collects results
The Keyframe System is Entirely Manual
  • .offload-base-commit is a checked-in file that someone manually updates
  • make_tar_of_repo.sh is a repo-specific script
  • generate_patch_for_offload.sh is a repo-specific script
  • The justfile recipe orchestrates all of this
  • When the base commit drifts too far, someone must update the SHA and rebuild the Docker image
Why This Exists

The keyframe pattern trades manual maintenance for speed. Instead of uploading the entire repo every run (Pattern A), only a small git diff patch is transferred. For large repos with many dependencies, this saves significant time during the prepare step. The checkpoint-images feature aims to automate this pattern.

C Default Provider Pattern

Config: [provider] type = "default" with custom prepare_command, create_command, etc.

Provider-agnostic: the shell commands can do anything.

Flow Steps

Same structure as Pattern A but fully configurable. Click to expand.

1
▶ offload run loads config once
Reads [provider] type = "default" and dispatches to DefaultProvider
  • Uses DefaultProvider::from_config() instead of ModalProvider::from_config()
  • All lifecycle commands are user-defined strings with placeholder variables

Example config:

prepare_command = "uv run @modal_sandbox.py prepare --include-cwd examples/Dockerfile" create_command = "uv run @modal_sandbox.py create {image_id}" exec_command = "uv run @modal_sandbox.py exec {sandbox_id} {command}" destroy_command = "uv run @modal_sandbox.py destroy {sandbox_id}" download_command = "uv run @modal_sandbox.py download {sandbox_id} {paths}"
↓
2
▶ prepare_command executes every run
User-defined command with flags appended by Rust

Rust appends standard flags to the user's prepare_command:

  • --cached (unless --no-cache was passed)
  • --copy-dir=local:remote for each configured/CLI copy directory
  • --sandbox-init-cmd=<quoted cmd> if set

The command must print an image ID as its last line of stdout.

Currently, all default-provider configs in use still point to @modal_sandbox.py, so the actual image build logic is the same as Pattern A. But the default provider is designed to be completely provider-agnostic.

↓
3
▶ create_command with {image_id} substitution parallel
Placeholder {image_id} replaced with prepare output
  • {image_id} is replaced with the string from prepare's stdout
  • Command must print a sandbox ID to stdout
  • Called N times to create the sandbox pool
↓
4
▶ exec_command / download_command / destroy_command
Standard lifecycle: run tests, collect results, tear down
  • exec_command: {sandbox_id} + {command} replaced
  • download_command: {sandbox_id} + {paths} replaced (for JUnit XML)
  • destroy_command: {sandbox_id} replaced

All placeholders are simple string substitution at runtime. No templating engine.

Design Intent

The default provider is the escape hatch. It allows Offload to work with any sandbox backend -- not just Modal. The commands are opaque shell strings; Offload does not interpret them. Image IDs and sandbox IDs are opaque strings.

Provider-Agnosticism Note

The checkpoint-images feature must work across all providers:

  • Modal provider: modal.Image.from_id(), modal.Image.from_dockerfile(), add_local_dir(), image IDs like im-XXXXXXXXX
  • Default provider: user-defined shell commands, image IDs are opaque strings
  • Local provider: no image concept at all
  • Any solution must not assume Modal-specific APIs at the Rust level

Image ID Expiry and Garbage Collection

How Modal image garbage collection interacts with the current caching system.

Cache File

.offload-image-cache

Stores a bare image ID string (e.g. im-XXXXXXXXX) representing the base Dockerfile image.

im-rlXozWoN3Q9TWD8I6fnxm5

This caches only the Dockerfile layer. The final image (with source + sandbox_init_cmd) is NOT cached.

Failure Scenarios

Scenario 1: Base image expired during prepare

The cached base image ID in .offload-image-cache was garbage-collected by Modal.

  1. _build_final_image() calls modal.Image.from_id(cached_id)
  2. Modal returns an error -- image no longer exists
  3. Python catches the exception
  4. Clears .offload-image-cache
  5. Rebuilds from scratch: modal.Image.from_dockerfile()
  6. Writes new image ID to .offload-image-cache
  7. Continues normally (one run is slower, subsequent runs are cached again)

Scenario 2: Final image expired during sandbox creation

The final image (output of prepare) was garbage-collected between prepare and create_sandbox.

  1. create_from_image(image_id) fails
  2. Logs error telling user to delete .offload-image-cache and re-run prepare
  3. This is a manual recovery -- there is NO automatic retry at this level
  4. In practice this is rare because the final image was just built seconds ago

Scenario 3: Keyframe pattern (Pattern B) resilience

The keyframe pattern is implicitly more resilient to GC.

  1. The tarball is baked into the Dockerfile, so the base image is always rebuildable
  2. Even if Modal GC's the cached image, rebuilding from the Dockerfile recreates it deterministically
  3. The trade-off is that the Dockerfile build is heavier (includes source + deps)
No Automatic Expiry Detection

There is no proactive expiry check. Failures surface only at runtime when a cached image ID is used. The system recovers gracefully for base images (Scenario 1) but requires manual intervention for final images (Scenario 2).

What Checkpoint Images Unify

The checkpoint-images feature makes Pattern B's optimization (small diff instead of full upload) available as a first-class Offload capability. Instead of manual scripts and pinned SHAs, offload checkpoint creates the keyframe and offload run automatically uses a thin diff layer. This eliminates the manual maintenance while preserving the performance benefit.

File Map

Key files involved in image building. Click each card for details.

Configuration Files

offload.toml
Primary config file for Offload. Defines provider, framework, groups, and report settings.

Loaded by offload run (or -c <path> for alternate configs).

Key fields for image building:

  • [provider] type -- selects provider implementation
  • [provider] dockerfile -- path to Dockerfile (modal)
  • [provider] include_cwd -- upload working dir (modal)
  • [offload] sandbox_init_cmd -- command baked into image
  • [provider] copy_dirs -- extra directories to include
.offload-image-cache
Caches the base Dockerfile image ID. Delete to force rebuild.

Contains a bare image ID string, e.g.:

im-rlXozWoN3Q9TWD8I6fnxm5

Written by modal_sandbox.py prepare. Read on subsequent runs to skip Dockerfile rebuild. Auto-cleared if image is GC'd.

Should be in .gitignore.

.offload-base-commit
Pattern B only. Contains pinned git SHA for the keyframe. Checked into repo.

Contains a bare git SHA, e.g.:

73ce8f3cc81924aba41ecf000524975b18979f6a

Manually maintained by developers. When updated, the Dockerfile must be rebuilt to include source at the new SHA.

Read by justfile recipe to generate tarball and patch.

.offload/checkpoint-cache
Future: checkpoint-images feature cache file. JSON with SHA and image ID.

Planned format:

{ "version": 1, "sha": "abc123def", "image_id": "im-XXXXXXXXX" }

Managed by offload checkpoint (create) and offload checkpoint --delete (remove).

When present, offload run uses diff-based image building instead of full rebuild.

Source Code

scripts/modal_sandbox.py
Bundled Python script for Modal sandbox lifecycle. Invoked via uv run @modal_sandbox.py.

CLI commands:

  • prepare <dockerfile> -- build image, return image_id
  • create <image_id> -- create sandbox, return sandbox_id
  • exec <sandbox_id> <command> -- run command in sandbox
  • destroy <sandbox_id> -- tear down sandbox
  • download <sandbox_id> <paths> -- download files from sandbox

Key functions: _build_base_image(), _build_final_image(), _read_image_cache(), _write_image_cache()

src/provider/modal.rs
Rust Modal provider implementation. Builds prepare command string, captures image_id.
  • ModalProvider::from_config() -- constructs the prepare command
  • Appends --include-cwd, --cached, --copy-dir, --sandbox-init-cmd
  • Captures image_id from stdout
  • Generates create/exec/destroy/download commands using the bundled script
src/provider/default.rs
Rust default provider. Runs user-defined shell commands with placeholder substitution.
  • DefaultProvider::from_config() -- reads command templates
  • Appends --cached, --copy-dir, --sandbox-init-cmd to prepare_command
  • Performs {image_id}, {sandbox_id}, {command}, {paths} substitution at runtime
src/provider.rs
SandboxProvider trait definition. Defines the provider interface.

Trait methods:

  • prepare() -- build/load image, return image_id
  • create_sandbox() -- create sandbox from image
  • exec_in_sandbox() -- run command in sandbox
  • destroy_sandbox() -- tear down sandbox
  • download_from_sandbox() -- download files

Checkpoint feature will add build_checkpoint() method.

Pattern B Scripts (Repo-Specific)

make_tar_of_repo.sh
Clones repo, checks out pinned SHA, creates tarball. Used by keyframe pattern.

Arguments: <sha> <tmpdir>

  • Clones current repo to temp directory
  • Checks out the specified SHA
  • Creates current.tar.gz
  • Copies tarball to CWD for Dockerfile COPY

This is a repo-specific script, not part of Offload itself.

generate_patch_for_offload.sh
Generates git diff against pinned SHA. Includes uncommitted changes.

Arguments: <sha>

  • Runs git diff <sha> against working tree
  • Includes uncommitted changes
  • Outputs patch to /tmp/<uuid>/patch

The patch file is passed to Offload via --copy-dir and applied by sandbox_init_cmd.
