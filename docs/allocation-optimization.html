What Changed and Why

Background: Rust ownership and allocation

Rust enforces strict ownership of data at compile time. When a function needs a string, it either borrows it (zero-cost, &str) or owns it (String, which heap-allocates). Offload’s orchestrator was written "ownership-first" for correctness — cloning strings at call boundaries even when the callee only reads them. This refactor tightens those boundaries without changing any public API.

Why 1.1 was abandoned

The “eliminate double allocation” premise was wrong. resolve_test_ids() requires &[String], so converting batch_ids to Vec<&str> still allocated N strings for that call plus an extra Vec<&str>. Net effect: +4.5% instruction regression. The original code was correct.

Code diff summary

7 files modified across src/. The total diff is ~43 lines. No public API changes; no new dependencies; all existing tests pass unmodified.

Representative change: returning a borrow instead of cloning (framework.rs) 
// Before: clones a String from the batch_ids slice
pub(crate) fn resolve_test_id_suffix_matching(
    name: &str,
    classname: Option<&str>,
    batch_ids: &[String],
) -> Result<String, String> {
    ...
    Ok(best_ids[0].clone())
}

// After: borrows from the caller's slice — zero allocation
pub(crate) fn resolve_test_id_suffix_matching<'a>(
    name: &str,
    classname: Option<&str>,
    batch_ids: &'a [String],
) -> Result<&'a str, String> {
    ...
    Ok(best_ids[0].as_str())
}

The lifetime 'a tells the compiler the returned &str lives as long as batch_ids. Callers that need an owned String add .to_string() at their site — the allocation moves to where it is actually needed.

Representative change: hoisting a loop-invariant allocation (vitest.rs) 
// Before: allocates file.to_string() once for suite name,
// then again per assertion in the loop
let mut suite = SuiteData {
    name: file.to_string(),  // alloc 1
    ...
};
for ar in &test_result.assertion_results {
    let classname = file.to_string();  // alloc 2..N
    ...
}

// After: one allocation, then clone (which may be cheaper
// for short strings due to small-string optimization)
let file_string = file.to_string();
let mut suite = SuiteData {
    name: file_string.clone(),
    ...
};
for ar in &test_result.assertion_results {
    let classname = file_string.clone();
    ...
}

Methodology and Detailed Results

Test protocol

  1. Build cargo build --release on each revision
  2. One warmup run (discarded)
  3. Five timed runs using /usr/bin/time -l on macOS (Apple Silicon)
  4. Sort by wall-clock time; discard the highest and lowest; report the middle three (same three runs for all metrics)
  5. Clean test-results/ and artifact-output/ between every run

Workload: offload-pytest-modal.toml — 18 pytest tests in 3 groups (unit with 2 retries, slow with 3 retries, flaky with 5 retries) executed in Modal remote sandboxes via uv run pytest. Sandboxes are built from examples/Dockerfile with the full working directory synced (include_cwd = true).

Revision control: Jujutsu (jj). Baseline = main (cda3cb08). Optimized = danver/optimize-instructions-1-2 (f8d1342e). Working copy switched with jj edit; release binary rebuilt from scratch for each version. Baseline runs executed first; optimized runs re-executed after a sandbox preparation bug fix.

Machine: macOS Darwin 25.2.0, Apple Silicon. Remote execution on Modal cloud infrastructure. No control over Modal’s container scheduling, cold-start behavior, or network conditions.

The static count understates the runtime effect because two of the removed call sites (vitest.rs loop, resolve_test_id_suffix_matching) execute per test case, not once. For a 500-test suite, those two changes alone eliminate ~1,000 runtime allocations.

Interpretation and caveats

  • All metrics are within noise. The Modal workload adds network latency, container cold-starts, and remote execution variance that dwarfs the ~43-line allocation changes. The Rust orchestrator’s CPU contribution (~4 s) is ~20% of the ~19 s wall clock; allocation overhead is a fraction of that.
  • Instructions (−3.6%) and cycles (−5.2%) show a directional trend consistent with fewer allocations, but run-to-run variance is high (±7%) and n=3 is too small for statistical significance. A dhat heap profiler or counting-allocator benchmark would isolate the effect.
  • CPU time is flat despite the instruction/cycle improvement, likely because the time saved in allocations is recouped by variance in Modal API calls, TLS handshakes, and async I/O scheduling.
  • n=3 is small. These results establish a direction, not a precise effect size. A rigorous evaluation would use 30+ runs with system isolation.
  • The primary value is code quality. Tighter ownership semantics, fewer unnecessary clones, and more idiomatic Rust APIs. The performance improvement is a bonus that would compound at larger workload scales (hundreds of tests, deep JUnit trees).
  • The abandoned optimization (1.1) is a useful negative result. It demonstrates that “fewer allocations” is not automatically faster when the downstream API forces re-allocation. Always profile.
