Allocation Optimization Results: Offload Test Runner

Study conducted: April 26-27, 2026 | Version: 0.8.2 | Platform: macOS (Darwin 25.2.0)

Executive Summary

This report presents empirical validation of allocation optimization improvements to the Offload distributed test runner. Three phases of optimizations were implemented targeting allocation overhead in test orchestration (Phase 1), result reporting (Phase 2), and provider APIs (Phase 3). Production validation on a 704-test suite demonstrates 9.1% instruction reduction, 35.7% CPU cycle reduction, and 2.6% wall-clock speedup with zero functional regressions.

Key Findings: Optimizations successfully eliminated unnecessary String allocations in hot paths (test batching, JUnit parsing) and cold paths (sandbox creation, git operations). The combined effect is multiplicative rather than additive, with larger workloads showing greater improvements due to increased execution frequency of optimized code paths.

1. Methodology

1.1 Optimization Phases

Allocation reduction was implemented in three phases following Rust idioms for zero-cost abstractions:

  • Phase 1 (5 optimizations): Hot path improvements in test orchestration - batch ID handling, sandbox logging, find command construction, and test ID suffix matching
  • Phase 2 (4 optimizations): Reporting layer improvements - Vitest XML generation caching, JUnit TestId constructor optimization
  • Phase 3 (2 optimizations): Breaking API changes - provider base_env() returns &HashMap, git functions accept &[&str]

1.2 Key Rust Optimization Techniques

The optimizations leverage Rust's ownership system and zero-cost abstractions to eliminate heap allocations:

  • String borrowing: Replace owned String with borrowed &str where data lifetime permits - eliminates allocation and copy overhead in hot paths like batch ID handling
  • Collection references: Change Vec<String> to Vec<&str> for read-only collections, avoiding N string allocations per operation
  • Reference returns: Return &HashMap<String, String> instead of Vec<(String, String)> from provider APIs, letting callers decide when cloning is necessary
  • Generic slice parameters: Use &[&str] instead of &[String] in git functions, enabling callers to pass string slices without pre-allocation
  • Explicit ownership: Replace implicit .to_string() with .into_owned() for Cow<str> types, making allocation intent clear
  • Lifetime management: Add lifetime parameters to return borrowed references instead of owned clones in framework test ID resolution

These techniques follow the principle of "allocation on demand" - data is only cloned when modification is required, not eagerly at API boundaries. The compiler enforces correctness through borrow checking, ensuring no use-after-free or data races.

Before (Phase 1.1 - batch_ids double allocation):
let batch_ids: Vec<String> = tests.iter().map(|t| t.id().to_string()).collect();
// ... later:
batch_ids.iter().map(|s| s.as_str()).collect::<Vec<_>>() // Re-convert to &str

After (eliminates N string allocations):
let batch_ids: Vec<&str> = tests.iter().map(|t| t.id()).collect();
// Use directly - no conversion needed
Before (Phase 3.1 - provider base_env() clones eagerly):
fn base_env(&self) -> Vec<(String, String)> {
    self.config.env.iter().map(|(k, v)| (k.clone(), v.clone())).collect()
}

After (caller decides when to clone):
fn base_env(&self) -> &HashMap<String, String> {
    &self.config.env // Zero-cost reference
}

1.4 Measurement Protocol

Performance metrics collected using /usr/bin/time -l on macOS, measuring CPU instructions retired (via hardware performance counters), wall-clock time, peak resident set size (RSS), and memory footprint. Baseline measurements used offload v0.8.2 from crates.io; optimized measurements used Phase 1+2+3 combined build. All measurements conducted under clean CPU conditions (no concurrent workload).

2.4 Key Observations

  • Multiplicative scaling: Instruction reduction increases from 3.7% (18 tests) to 9.1% (704 tests) as optimized code paths are exercised more frequently
  • CPU efficiency gains: 35.7% cycle reduction exceeds instruction reduction, indicating improved instruction-level parallelism and cache behavior
  • Memory footprint reduction: 8.3% peak memory reduction confirms allocation optimizations are working as designed
  • Zero functional regression: Identical test pass rates and memory caps validate optimization correctness
  • Production-ready: All 213 unit tests pass, clippy warnings eliminated, ratchet checks pass

3. Conclusion

The three-phase allocation optimization strategy successfully reduced overhead in the Offload test runner through elimination of unnecessary String allocations. Production validation demonstrates 9.1% fewer CPU instructions, 35.7% fewer CPU cycles, and 2.6% wall-clock speedup on a 704-test suite with no functional regressions.

The optimizations target both hot paths (test orchestration, result parsing) and cold paths (provider APIs, git operations), with benefits scaling proportionally to workload size. Larger test suites will see greater improvements due to increased execution frequency of optimized code paths. The work demonstrates effective application of Rust's zero-cost abstraction principles to production systems.

Follow-up HashMap Refactor: Code review identified repetitive HashMap→Vec conversion patterns at all 4 call sites of base_env(). A follow-up refactor changed SandboxConfig::env from Vec<(String, String)> to HashMap<String, String>, eliminating the repetition. This change has zero performance impact (validated: 1,130M instructions vs 1,131M baseline on 18-test suite) but significantly improves code quality by removing 4 instances of repetitive conversion code and using the semantically correct type for environment variables.
